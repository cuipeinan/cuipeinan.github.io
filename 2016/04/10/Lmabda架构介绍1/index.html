Lmabda架构介绍（一）

发表于 2016-04-10   |   分类于 Lambda架构   |  

摘要：

        Lmabda架构介绍

正文：

实时计算任意数据集上的任意函数，是一个令人望而却步的问题。没有单独的工具可以提供完整的解决方案。相反，你必须使用各种工具和技术构建一个完整的大数据系统。

Lambda架构的主要思想是将大数据系统建立为一系列的层，如图1.6所示。每一层满足属性的一个子集，且通过该层的下一层提供的功能来构建。你会从整本书中学习如何设计、实现和部署每一层，但整个系统是如何有机结合在一起的高层思想是很容易理解的。

Lmabda架构分为3层，批处理层，服务层和速度层：

  • Batch Layer
  • Serving Layer
  • Speed Layer

1.Batch Layer

批处理层生成批处理视图。需要能够做俩件事：存储不可变的，不断增长的主数据集；在该数据集上运算任意函数。Hadoop是批处理系统的一个典型例子，我们将使用Hadoop来阐述批处理层的概念。

2.Serving Layer

服务层加载批处理层生成的批处理视图，这样它们才能被查询到。服务层是一个专用的分布式数据库，用于加载批处理视图，并可以对它进行随机读取。当新的批处理视图可用时，服务层自动替换哪些视图，这样更多的最新结果就是可用的了。

服务层支持随机读取批处理视图和支持批处理更新。最值得注意的是，它不需要支持随机写操作，因为随机写操作会再数据库中导致绝大多数的复杂性。因为不支持随机写，所以这些数据库是非常简单的。简单性使得他们是健壮的，可预测的，易于配置的，且操作简单的ElephantDB，是Lambda中使用的服务层数据库，它只有几千行代码。

3.Speed Layer

当批处理层完成预先计算批处理视图时，服务层就进行更新。这意味着唯一没有在批处理视图中展示的数据，是运行预先计算期间新来的数据。速度层的目的，实现一个完全实时数据系统剩下的数据要完成（即为了实时在任意数据上执行任意函数计算–弥补最后几个小时的哪些数据。），顾名思义，它的目标是确保新的数据按照应用程序需求尽快展示在查询函数中。

速度层层做三件事：

  1. 补偿服务层高潜伏的更新。
  2. 首先增量计算
  3. Batch layer 最终覆盖速度层。

速度层类似批处理层，他基于接收到的数据生成视图。他们有俩大区别：

  1. 速度层只查看最近的数据，而批处理层要立即查看所有数据。
  2. 为了达到最小延迟的可能，速度层不会立即查看所有的新数据。 相反， 每当接受到新的数据，它就更新实时视图，而不是像批处理层从头开始重新计算视图。速度层做增量计算，而不是像批处理层中进行重新计算。

总结

Lambda架构总体被总结为这三个公式：

batch view = function(all data);   
realtime view = function (realtime view,new data);
query = function (batch view,realtime view);
